The 7 Pillars

Every aspect of pi configuration maps to one of seven foundational pillars. Together they form a comprehensive harness.

7
Pillars
21
Max Score
13
Atomic Skills
1

Skill Composition

→ pi-skills

"What should pi be able to do?"

What atomic skills and composed workflows the project needs. Skills capture complex workflows so pi can execute them reliably without re-explanation.

What Gets Scored

  • Project-specific skills or commands exist
  • Skills have clear triggers and bounded scope
  • Skill composition chains atomic skills together
2

Context Engineering

→ pi-context, pi-scaffold

"What does pi need to know?"

What information pi loads and when. Context budget analysis, prompt organization, and progressive disclosure design.

Context Layers

  • Always-loaded prompts (project.md, style.md)
  • On-demand prompts (deploy.md, migrate.md)
  • Discovered at runtime (file contents, git history)
  • Persistent memory (user, feedback, project)
3

Orchestration & Routing

→ pi-routing

"How should work be distributed?"

How tasks route to specialist agents. Parallel vs serialize decisions, subagent delegation patterns, and build-loop workflows.

Decision Matrix

  • Subagent for independent, expensive tasks
  • Inline for trivial tasks or tight iteration
  • Parallel for independent research queries
  • Serialize for state-changing sequences
4

Persistence & State

→ pi-memory

"What should pi remember?"

Cross-session memory and knowledge. What persists, what doesn't, and how memories are organized and maintained.

Memory Types

  • User - role, preferences, expertise level
  • Feedback - corrections AND confirmations
  • Project - ongoing work, decisions, deadlines
  • Reference - external systems, URLs
5

Quality Gates

→ pi-gates, pi-hooks

"How do we catch mistakes early?"

Automated validation and feedback loops. Post-edit checks, pre-commit gates, and conversation-end drift detection.

Gate Layers

  • Post-edit: Format, lint, parse validation
  • Pre-commit: Changed-file checks (<10s)
  • Pre-push: Full test suite (<2min)
  • Conversation-end: Harness health check
6

Permissions & Safety

→ pi-permissions

"What can pi do without asking?"

Permission boundaries: auto-allow safe operations, gate dangerous ones. Balance flow with guardrails.

Blast Radius Principle

  • Local + reversible = auto-allow (read, format, test)
  • Local + hard-to-reverse = prompt (delete, reset)
  • Shared state = always prompt (push, deploy)
  • External systems = always prompt (API calls)
7

Ergonomics & Trust

→ pi-ergonomics

"How should pi communicate?"

Interaction style and trust calibration. Verbosity, autonomy level, output format, and feedback capture.

Trust Levels

  • Level 1: New user - confirm everything, verbose
  • Level 2: Established - routine auto, confirm destructive
  • Level 3: Power user - autonomous, terse
